# Описание решения

Код решения: [github.com/kite121/avito-retriever](https://github.com/kite121/avito-retriever).

Основная воспроизводимая точка входа — `python tools/make_answer.py` (или
`make answer`). Она читает запросы из `test.f`, применяет одну фиксированную
архитектуру и создаёт `answer.csv`, не запуская перебор параметров и не используя
разметку test-набора.

## Данные и постановка

В работе использованы три Feather-файла:

| Файл | Строк | Назначение |
|---|---:|---|
| `articles.f` | 793 | `article_id`, заголовок и HTML статьи |
| `calibration.f` | 500 | запросы с известными релевантными `article_id` |
| `test.f` | 500 | запросы, для которых строится `answer.csv` |

В `calibration.f` на один запрос приходится в среднем 1,522 релевантной статьи,
максимум — 4; всего в разметке встречается 79 уникальных релевантных статей.
Целевая метрика — MAP@10. Ответ для каждого запроса содержит не более десяти
уникальных идентификаторов статей в порядке убывания релевантности.

## Обработка HTML статей

HTML не преобразуется в один плоский текст. Парсер на BeautifulSoup + `lxml`
выделяет отдельные поисковые поля:

- `title` — заголовок из таблицы статей;
- `headings` — `h1`–`h3`, нестандартные `headline` и заголовки `chunk`;
- `ui_labels` — подписи вкладок и спойлеров (`tab-label`, `spoiler-text`);
- `body` — основной текст после удаления уже выделенных структурных блоков;
- `tables` — строки таблиц с разделением ячеек через `|`;
- `image_alt` — полезные `alt`/`title` изображений без имён файлов и служебных
  подписей вроде «иконка»;
- `image_ocr` — отдельное поле для опционального OCR.

Перед извлечением текста удаляются `script`, `style`, `noscript`, `svg` и
`input`. Тексты внутри каждого поля нормализуются и дедуплицируются. Структурные
элементы удаляются из копии дерева перед сборкой `body`, поэтому заголовки,
таблицы и UI-подписи не учитываются дважды.

Лексическая нормализация включает Unicode NFKC, нижний регистр, замену `ё` на
`е`, унификацию кавычек и тире, а также схлопывание пробелов. При этом исходная
структура полей сохраняется для разных весов BM25F.

Аудит корпуса показал, что основной текст есть у 792 из 793 статей, заголовки
разделов — у 476, UI-подписи — у 450, таблицы — у 67, полезный `image_alt` — у
241. В 373 статьях найдено 1708 изображений. OCR предусмотрен как отдельная
ablation, но в конфигурации отправленного запуска отключён: `make answer`
воспроизводит вариант без OCR. Это исключает сетевые ошибки загрузки картинок и
зависимость результата от версии OCR-модели.

## Признаки, модели и алгоритмы

Итоговое ранжирование объединяет взаимодополняющие сигналы.

### 1. SentencePiece BM25F

SentencePiece Unigram с размером словаря 8000 обучается на полях статей и
запросах `calibration.f`. Субсловная токенизация устойчивее обычных слов к
опечаткам, окончаниям и слитному написанию.

BM25F индексирует поля раздельно. В фиксированной конфигурации веса равны:

| Поле | Вес |
|---|---:|
| `title` | 5,0 |
| `headings` | 3,0 |
| `ui_labels` | 2,5 |
| `body` | 1,0 |
| `tables` | 1,0 |
| `image_alt` | 0,8 |
| `image_ocr` | 0,5 (поле пустое при запуске без OCR) |

Так короткие названия и заголовки разделов не теряются на фоне длинного body.
Для каждого query сохраняется top-100 лексических кандидатов.

### 2. Dense retrieval по фрагментам

Статьи разбиваются на фрагменты по 256 слов с перекрытием 48. К каждому фрагменту
добавляются title, headings и UI labels. Модель
`intfloat/multilingual-e5-small` кодирует запросы с префиксом `query:`, а
фрагменты — с `passage:`. Эмбеддинги L2-нормализуются; оценка статьи равна
максимальному cosine similarity среди её фрагментов. Это добавляет семантическое
сопоставление перефразированных запросов и не размывает сигнал длинной статьи.

### 3. kNN по размеченным calibration-запросам

Для test-query ищутся похожие запросы из `calibration.f` двумя способами:
SentencePiece-BM25 и cosine similarity E5. Списки соседей объединяются RRF,
после чего их ground-truth статьи получают взвешенные голоса. Используется top-10
соседей. Этот канал переносит устойчивые соответствия «формулировка запроса →
статья» без обучения отдельного классификатора на 500 строках.

При оценке на `calibration.f` kNN строится строго out-of-fold: запрос никогда не
может выбрать сам себя или точный нормализованный дубликат из своего fold.

### 4. Fusion и reranking

Top-100 от BM25F, dense и kNN объединяются weighted Reciprocal Rank Fusion:
веса `1,0 / 1,0 / 0,7`, `rrf_k = 40`. RRF использует позиции, а не несопоставимые
сырые оценки разных моделей.

Из объединённого списка берутся top-20 кандидатов. Cross-encoder
`BAAI/bge-reranker-v2-m3` оценивает пары «query — лучший фрагмент статьи» и
переставляет кандидатов. В `answer.csv` попадают первые 10 уникальных статей.

## Проверка качества на calibration.f

`calibration.f` делится на пять детерминированных fold с `seed = 42`. Точные
нормализованные дубликаты запросов всегда относятся к одному fold, чтобы одна и
та же формулировка не оказалась одновременно в обучающей и проверочной части.

Один fold заранее резервируется как `confirm`, остальные четыре используются
как `tune`. Порядок работы был таким:

1. На tune выбирались SentencePiece/BM25F, dense-модель, размер фрагмента,
   параметры kNN, веса RRF и reranker.
2. Все промежуточные ранжирования сохранялись в едином формате
   `query_id, article_id, score, rank, source`.
3. Считались AP@10 по каждому query, итоговый MAP@10 и candidate recall@20/50/100.
4. После фиксации параметров архитектуры сравнивались только на untouched confirm.
5. Для парных разностей AP@10 использовались bootstrap confidence interval и
   permutation test (по 10 000 повторов), а множественные сравнения
   корректировались методом Holm–Bonferroni.
6. После выбора архитектуры все 500 calibration-query используются для обучения
   SentencePiece и как база размеченных соседей при финальном предсказании test;
   ground truth не участвует в BM25F, dense retrieval или reranking напрямую.

Контрольный воспроизводимый BM25F baseline на всех 500 calibration-query дал:

| Метрика | Значение |
|---|---:|
| MAP@10 | 0,279696 |
| Recall@20 | 0,693833 |
| Recall@50 | 0,819333 |
| Recall@100 | 0,893667 |

Эти числа сохранены в `artifacts/runs/.../metrics.json` и служат sanity check
оценочного кода. Подбор и сравнение расширенной архитектуры вынесены в ноутбуки
`01`–`05`; production-команда их не повторяет и читает фиксированные параметры.

## Анализ ошибок и изменения в решении

У BM25F baseline 173 из 500 запросов имели AP@10 = 0; для 41 запроса релевантная
статья не попадала даже в top-100. Разбор запросов и найденных статей показал
следующие классы ошибок.

| Тип ошибки | Наблюдение | Что сделано |
|---|---|---|
| Потеря структуры HTML | При простом `get_text()` UI-подписи, таблицы и заголовки смешивались с body, а важные короткие фразы терялись | Структурные поля извлекаются отдельно, дедуплицируются и получают разные веса BM25F |
| Опечатки и слитные слова | Встречаются `Отмменит`, `безопастно`, `авитодоставкой`, текст без пробела после запятой | NFKC-нормализация и SentencePiece вместо только словной токенизации |
| Разговорный шум и эмоции | «Здравствуйте», «подскажите», «вы оборзели со стоимостью доставки» дают слабое буквальное совпадение с заголовком справки | Dense E5 добавляет семантический сигнал; fusion не позволяет одному шумному каналу доминировать |
| Перефразы одной причины обращения | «Почему такая дорогая доставка?» должно вести к статье «Кто оплачивает доставку и сколько она стоит» | Dense retrieval и lexical+dense kNN по похожим размеченным обращениям |
| Неоднозначные платежи и возвраты | Формулировки «деньги списали» или «когда вернут деньги» соответствуют разным сценариям: объявление, доставка, отмена, двойное списание | Сначала собирается широкий top-100 из трёх каналов, затем cross-encoder различает контекст внутри top-20 |
| Длинные статьи и локальный ответ | Релевантный абзац может занимать малую часть большой статьи | Перекрывающиеся чанки и max-score по лучшему фрагменту; reranker получает этот фрагмент, а не весь HTML |
| Картинки и нестабильный OCR | Часть инструкций содержит изображения, но загрузки могут падать, а OCR добавляет шум | Полезные alt-тексты используются всегда; OCR оставлен отдельным ablation и отключён в воспроизводимом финальном запуске |
| Кандидат найден, но стоит низко | Высокий recall@100 baseline не гарантирует высокий MAP@10 | Cross-encoder переставляет небольшой единый пул кандидатов непосредственно по релевантности пары query–article |

После анализа также добавлены контрактные проверки: нельзя сохранить ответ с
пропущенным или повторным `query_id`, пустым списком, повторяющимися или
неизвестными `article_id`, а также более чем 10 статьями.

## Воспроизведение отправленного answer.csv

Из корня чистого клона:

```bash
python3 -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
python -m pip install -e ".[lexical,neural,dev]"
python tools/make_answer.py
```

Используются включённые в репозиторий входные файлы со следующими SHA-256:

```text
articles.f     2f585ef7a4a9568412efc7cb30f210b122aed7dc57b893ba09774b8be7185ea7
calibration.f  efc6c58a121775095fbed5c9aeb30aa65165b4e3c33da2acbf839688ff719d72
test.f         d9857c0286a710d0fd43c0b1ce98564a4144463336b3a8d16be97ccf0ffd45cf
```

Пайплайн вычисляет fingerprint из содержимого этих файлов, полного фиксированного
конфига и флага OCR. Артефакты другого прогона не могут быть случайно
переиспользованы. Финальное ранжирование кешируется, а `answer.csv` каждый раз
собирается заново в порядке `query_id` и `rank` и проходит встроенный валидатор.

Для отправленного варианта команда должна запускаться без `--ocr` и без изменения
`configs/experiments/fixed_submission.yaml`. Флаг `--force` допустим: он не меняет
состав пайплайна, а только игнорирует кеш финального ranking и пересчитывает те же
этапы.

Дополнительная ручная проверка:

```bash
avito-validate-submission answer.csv --data-dir data/candidate_data
```

Основные файлы реализации:

- `tools/make_answer.py` — CLI для воспроизведения;
- `src/avito_retriever/pipeline/fixed.py` — полный pipeline и сборка CSV;
- `src/avito_retriever/preprocessing/html.py` — HTML-парсер;
- `src/avito_retriever/retrieval/` — BM25F, dense и kNN;
- `src/avito_retriever/fusion/rrf.py` — объединение списков;
- `src/avito_retriever/reranking/cross_encoder.py` — финальный reranking;
- `configs/experiments/fixed_submission.yaml` — все параметры результата.
